4. МОДЕЛЬ ДАННЫХ

4.2. Описание сущностей

Справочные таблицы

Таблица "статусы_турниров" — справочная таблица для нормализации статусов турниров:

  • id (SERIAL PRIMARY KEY) — уникальный идентификатор;
  • код (VARCHAR(20) NOT NULL UNIQUE) — код статуса ('upcoming', 'active', 'completed');
  • название (VARCHAR(100) NOT NULL UNIQUE) — название статуса ('Предстоящий', 'Активный', 'Завершен');
  • описание (TEXT) — описание статуса;
  • дата_создания (TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL) — дата создания записи.

Таблица "позиции_дебатов" — справочная таблица для нормализации позиций в дебатах:

  • id (SERIAL PRIMARY KEY) — уникальный идентификатор;
  • код (VARCHAR(10) NOT NULL UNIQUE) — код позиции ('for', 'against');
  • название (VARCHAR(50) NOT NULL UNIQUE) — название позиции ('За', 'Против');
  • описание (TEXT) — описание позиции;
  • дата_создания (TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL) — дата создания записи.

Основные таблицы

Таблица "участники" — хранит информацию об участниках дискуссионного клуба:

  • id (SERIAL PRIMARY KEY) — уникальный идентификатор;
  • имя (VARCHAR(100) NOT NULL) — имя участника;
  • фамилия (VARCHAR(100) NOT NULL) — фамилия участника;
  • электронная_почта (VARCHAR(255) NOT NULL UNIQUE) — электронная почта (уникальная);
  • дата_создания (TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL) — дата создания записи.

Таблица "жюри" — хранит информацию о членах жюри:

  • id (SERIAL PRIMARY KEY) — уникальный идентификатор;
  • имя (VARCHAR(100) NOT NULL) — имя члена жюри;
  • фамилия (VARCHAR(100) NOT NULL) — фамилия члена жюри;
  • электронная_почта (VARCHAR(255) NOT NULL UNIQUE) — электронная почта (уникальная);
  • дата_создания (TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL) — дата создания записи.

Таблица "темы" — хранит темы для обсуждения в дебатах:

  • id (SERIAL PRIMARY KEY) — уникальный идентификатор;
  • заголовок (VARCHAR(500) NOT NULL) — заголовок темы;
  • описание (TEXT) — описание темы;
  • дата_создания (TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL) — дата создания записи.

Таблица "сезоны" — хранит информацию о временных периодах проведения турниров:

  • id (SERIAL PRIMARY KEY) — уникальный идентификатор;
  • название (VARCHAR(255) NOT NULL) — название сезона;
  • дата_начала (DATE NOT NULL) — дата начала сезона;
  • дата_окончания (DATE NOT NULL) — дата окончания сезона;
  • дата_создания (TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL) — дата создания записи;
  • CONSTRAINT сезоны_даты_валидны CHECK (дата_окончания >= дата_начала) — проверка корректности дат.

Таблица "турниры" — хранит информацию о турнирах в рамках сезонов:

  • id (SERIAL PRIMARY KEY) — уникальный идентификатор;
  • ид_сезона (INTEGER NOT NULL REFERENCES сезоны(id)) — внешний ключ на таблицу сезонов;
  • название (VARCHAR(200) NOT NULL) — название турнира;
  • дата_начала (DATE NOT NULL) — дата начала турнира;
  • дата_окончания (DATE) — дата окончания турнира (может быть NULL);
  • ид_статуса (INTEGER NOT NULL REFERENCES статусы_турниров(id)) — внешний ключ на таблицу статусов турниров;
  • дата_создания (TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL) — дата создания записи;
  • CONSTRAINT турниры_даты_валидны CHECK (дата_окончания IS NULL OR дата_окончания >= дата_начала) — проверка корректности дат.

Таблица "раунды" — хранит информацию об отдельных раундах дебатов в турнирах:

  • id (SERIAL PRIMARY KEY) — уникальный идентификатор;
  • ид_турнира (INTEGER NOT NULL REFERENCES турниры(id)) — внешний ключ на таблицу турниров;
  • ид_темы (INTEGER NOT NULL REFERENCES темы(id)) — внешний ключ на таблицу тем;
  • номер_раунда (INTEGER NOT NULL) — номер раунда в турнире;
  • дата_раунда (DATE NOT NULL) — дата проведения раунда;
  • дата_создания (TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL) — дата создания записи;
  • CONSTRAINT раунды_уникальный_турнир_номер UNIQUE (ид_турнира, номер_раунда) — уникальность номера раунда в рамках турнира;
  • CONSTRAINT раунды_номер_положительный CHECK (номер_раунда > 0) — проверка положительности номера раунда.

Таблица "выступления" — хранит записи о выступлениях участников с оценками:

  • id (SERIAL PRIMARY KEY) — уникальный идентификатор;
  • ид_раунда (INTEGER NOT NULL REFERENCES раунды(id)) — внешний ключ на таблицу раундов;
  • ид_участника (INTEGER NOT NULL REFERENCES участники(id)) — внешний ключ на таблицу участников;
  • ид_позиции (INTEGER NOT NULL REFERENCES позиции_дебатов(id)) — внешний ключ на таблицу позиций дебатов;
  • оценка_логики (INTEGER CHECK (оценка_логики IS NULL OR (оценка_логики >= 1 AND оценка_логики <= 10))) — оценка по критерию логики (от 1 до 10, может быть NULL);
  • оценка_риторики (INTEGER CHECK (оценка_риторики IS NULL OR (оценка_риторики >= 1 AND оценка_риторики <= 10))) — оценка по критерию риторики (от 1 до 10, может быть NULL);
  • оценка_эрудиции (INTEGER CHECK (оценка_эрудиции IS NULL OR (оценка_эрудиции >= 1 AND оценка_эрудиции <= 10))) — оценка по критерию эрудиции (от 1 до 10, может быть NULL);
  • ид_судьи (INTEGER NOT NULL REFERENCES жюри(id)) — внешний ключ на таблицу жюри;
  • дата_создания (TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL) — дата создания записи;
  • CONSTRAINT выступления_уникальный_раунд_участник UNIQUE (ид_раунда, ид_участника) — уникальность выступления участника в раунде.

Таблицы аудита

Таблица "аудит_участников" — хранит историю изменений данных участников:

  • id (SERIAL PRIMARY KEY) — уникальный идентификатор;
  • ид_участника (INTEGER NOT NULL REFERENCES участники(id)) — внешний ключ на таблицу участников;
  • действие (VARCHAR(10) NOT NULL CHECK (действие IN ('INSERT', 'UPDATE', 'DELETE'))) — тип действия (INSERT, UPDATE, DELETE);
  • дата_изменения (TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL) — дата и время изменения;
  • старые_данные (JSONB) — данные до изменения в формате JSON;
  • новые_данные (JSONB) — данные после изменения в формате JSON.

4.3. Нормализация

База данных нормализована до BCNF (Boyce-Codd Normal Form):

  • Каждая таблица имеет первичный ключ (id);
  • Все атрибуты полностью зависят от первичного ключа;
  • Нет транзитивных зависимостей;
  • Все детерминанты являются потенциальными ключами.

Для обеспечения нормализации были созданы справочные таблицы:

  • статусы_турниров — вынесены статусы турниров из таблицы турниров для устранения избыточности данных;
  • позиции_дебатов — вынесены позиции в дебатах из таблицы выступлений для устранения избыточности данных.

Все внешние ключи настроены с соответствующими правилами каскадирования:

  • ON DELETE RESTRICT — предотвращает удаление связанных записей;
  • ON DELETE CASCADE — удаляет связанные записи при удалении родительской;
  • ON UPDATE CASCADE — обновляет связанные записи при обновлении родительской.
